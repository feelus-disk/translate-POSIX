1. Введение

1.1 Возможности(scope)

POSIX.1-2008 определяет стандартный интерфейс и среду операционной системы, включая командный интерпретатор (или "оболочку(shell)"), и общие программы-утилиты для поддержки портабельности приложений на уровне исходного кода. Предполагается,что он будет использоватся и разработчиками приложений и реализаторами системы.

POSIX.1-2008 содержит четыре главных компонента (каждый в соответствующей главе):

  1. Основные положения, концепции, и интерфейсы общие для всех глав POSIX.1-2008, включая соглашения утилит и и определения хедеров языка C, включены в главу POSIX.1-2008 Базовые определения.

  2. Определения для функций и подпрограмм системных сервисов, языко-зависимых системных сервисов для языка программирования C, проблемы функций, включая портабельность, обработку ошибок, и восстановление после ошибок, включены в главу POSIX.1-2008 системный интерфейс.

  3. Определения для стандартного интерфейса уровня исходного кода для сервисов интерпретации команд ("оболочки(shell)") и общих программ-утилит для прикладных программ включены в главу POSIX.1-2008 оболочка и утилиты.

  4. Расширенно пояснение, которое не вошло в другую часть структуры документа, содержит историческую информацию касательно концепций POSIX.1-2008 и почему фичи включены или исключены разработчиками стандарта, включены в главу POSIX.1-2008 Пояснение.

Следующие области не вошли в POSIX.1-2008:

  • Графический интерфейс

  • Интерфейс систем управления базами данных

  • Record I/O considerations

  • Переносимость объектного и бинарного кода

  • Конфигурация системы и доступность ресурсов

POSIX.1-2008 описывает внешние характеристики и возможности, которые важны для разработчиков приложений, а не конструкционные техники, требующиеся для достижения этих возможностей. Специальный акцент делается на те функции и возможности которые требуются для шрокого спектра коммерческих приложений.

Возможности, предоставляемые POSIX.1-2008 взяты из следующих базовых документов:

  • IEEE Std 1003.1, 2004 Edition (POSIX-1) (совместно с IEEE Std 1003.1-2001, IEEE Std 1003.1-2001/Cor 1-2002, и IEEE Std 1003.1-2001/Cor 2-2004)

  • The Open Group Technical Standard, 2006, Extended API Set Part 1

  • The Open Group Technical Standard, 2006, Extended API Set Part 2

  • The Open Group Technical Standard, 2006, Extended API Set Part 3

  • The Open Group Technical Standard, 2006, Extended API Set Part 4

  • ISO/IEC 9899:1999, Programming Languages - C, including ISO/IEC 9899:1999/Cor.1:2001(E), ISO/IEC 9899:1999/Cor.2:2004(E), and ISO/IEC 9899:1999/Cor.3

Акцент был сделан на том, чтобы стандартизировать существующие практики для существующих пользователей, с изменениями и дополнениями ограниченными корректировкой дефектов в следующих областях:

  • Проблема инициированная Austin Group defect reports, IEEE Interpretations against IEEE Std 1003.1, и ISO/IEC defect reports соместно с ISO/IEC 9945

  • Проблема инициированная опечаткой(corrigenda) The Open Group Technical Standards and working group resolutions from The Open Group

  • Проблема вызванная ISO TR 24715:2006, Конфликты между POSIX и LSB

  • Изменения, чтобы сделать текст самосогласованным с добавленным дополнительным материалом.

  • Фичи, помеченные как для совместимости ии устаревшие в базовых документах, было решено убрать из этой версии

  • Пересмотр и реорганизация опций в стандарте

  • Выровнено с ISO/IEC 9899:1999 стандартом, включая ISO/IEC 9899:1999/Cor.2:2004(E)

1.5 Терминология

Для целей POSIX.1-2008, применяется следующая терминология:

can

Описывается допустимая опциональная фича или поведение доступная пользователю или приложению. Фича или поведение является обязательной для реалзаций, соответствующих POSIX.1-2008. Приложение может полагатся на существование фичи или поведения.

зависит от реализации (implementation-defined)

Описывается величина или поведение, которое не определяется POSIX.1-2008, а выбирается реализатором. Величина или поведение может различатся на разных реализациях, соответствующих POSIX.1-2008. Приложение не должно полагаться на существование величины или поведения. Приложение, которое полагается на такую величину или поведение, не может прзнано портируемым между реализациями, соответствующими стандарту.

Реализатор должен документировать такую величину или поведение, чтобы оно могло корректно использоваться приложениями.

для совместимости(legacy)

Описывается фича или поведение, которое сохранено для совместимости со старыми приложениями, но которое имеет ограничение, делающее его неподходящим для разработки портируемых приложений. Новые приложения должны использовать альтернативные способы получения этой функциональности.

может(may)

Описывается фича или поведение, которое является опциональным для реализации, которая соответствует POSIX.1-2008. Приложения не должны полагаться на существование этой фичи или поведения. Приложение, которое полагается на такую фичу или поведение не может быть признано портируемым между реализациями, кторые соответствуют.

Чтобы избежать неоднозначность, в противоположность может(may) используется не требуется(need not), вместо не может(may not).

должен(shall)

Для реализаций, которые соответствуют POSIX.1-2008, описывается фича или поведение, которое обязательно. Приложение может полагаться на существование этой фичи или поведения.

Для приложения или пользователя, описывается поведение, которое обязательно.

желательно(should)

Для реализаций, которые соответствуют POSIX.1-2008, описывается фича или поведение, которое реомендуется но не обязатльно. Прилоежения не должны полагатьсяна существование этой фичи или поведения. Приложение, которое полагается на такую фичу или поведение не может быть признано портируемым между реализациями, кторые соответствуют.

Для прилоежний, описывается фича или поведение, которое является рекомендуемой практикой программирования для оптимальной портируемости.

неопределено(undefined)

Описывается природа значения или поведения, которое не определено стандартом POSIX.1-2008, которое является результатом использования невалидной программой конструкции невалидных входных данных.

Значение или поведение пожет различаться в зависимости от реализации, соответствующей POSIX.1-2008. Приложение не должно полагаться на существование или значимость значения или поведения. Приложение, которое полагается на какое-либо конкретное значение или поведение не может быть уверенно портируемым на соответствующие реализации.

незадано(unspecified)

Описывается природа значения или поведения не занного POSIX.1-2008, которое является результатом использования валидной конструкции программы или валидных входных данных.

Значение или поведение пожет различаться в зависимости от реализации, соответствующей POSIX.1-2008. Приложение не должно полагаться на существование или значимость значения или поведения. Приложение, которое полагается на какое-либо конкретное значение или поведение не может быть уверенно портируемым на соответствующие реализации.

1.7 Портируемость

Некоторые утилиты в главе POSIX.1-2008 Оболочка и утилиты и функции в главе POSIX.1-2008 Системный интерфейс описывают функциональность, которая может не быть польностью портируемой на системы, встречающие требования соответствия POSIX (см. Соответствие).

Где дополнительная, расширенная, или урезанная функциональность задается затененным текстом и кодом на полях, определяющим природу опции, расширения или предупреждения (см. Коды). Для максимальной портируемости приложение должно избегать такой функциональности.

Если основной задачей утилиты на является производство текстуальных материалов на ее стандартном выводе, разработчики приложения не должны полагаться на формат или содержание таких материалов. Когда основной задачей является производство таких материалов, но формат вывода не полностью задан, помечается кодом отступа OF и затеняется. Разработчики приложения предупреждены, что не стоит ожидать что вывод такого интерфейса на одной системе будет таким жа на другой системе.

1.7.2 нотация кодов отступа

Некоторая функциональность, описанная в POSIX.1-2008 зависит от поддержки более чем одной опции, или независимо может зависеть от нескольких опций. Следующая нотация кодов отступа используется для обозначения следующих случаев.

Возможность зависит от одной или двух опций

В этом случае, коды отступа имеют разделитель <пробел>; К примеру:

[SHM] Эта фича требует поддержки только опции объектов совместно используемой памяти (Shared Memory).

[SHM TYM] Эта возможность требует поддержки опции объектов совместно используемой памяти Shared Memory и опции объектов типизированной памяти (Typed Memory); это значит, что приложение, использующую эту возможность портируемо только между реализациями, которые предоставляют обе опции.

Возможности, зависящие от одной из двух перечисленных опций

В этом случае коды отступа имеют разделитель '|' обозначающий логическое OR; К примеру:

[SHM|TYM] Эта возможность зависит от отдной из двух опций Shared Memory Objects option or the Typed Memory Objects option; that is, an application which uses this feature is portable between implementations that provide any (or all) of the options.

Аозможность зависит от более чем двух опций

Используется следующая сокращенная нотация:

[MC1] Код отступа MC1 является сокращением для TPP|TPI|RPP|RPI. Возможности, которые затенеты этип кодом оступа требуют поддержки одной из Non-Robust Mutex Priority Protection option или the Non-Robust Mutex Priority Inheritance option или the Robust Mutex Priority Protection option или the Robust Mutex Priority Inheritance option.

Большие секции, зависящие от опции

Где большие разделы завися от поддержки некоторой опции, предоставляется и затеняется ведущий блок текста; к примеру:

[XSI] Этот раздел описывает расширение для поддержки межпроцессной коммуникации. Функциональность, описанная в этом разделе должна предоставляться на реализациях, которые поддерживают опцию XSI (и оставшаяся часть этого раздела не отмечена).
